# Clases

La razón es que las clases en JavaScript (**class**) son simplemente "syntactic sugar" (azúcar sintáctica) sobre el sistema de prototipos.

```javascript

class PersonClass {
    constructor(name, age) {
        this.name = name;
        this.age = age;
    }

    showAge() {
        console.log(`Age of ${this.name} is ${this.age}.`);
}
}

const pedro = new PersonClass('Pedro', 30);
console.log(pedro)
pedro.showAge(); // Output: Age of Pedro is 30.

```

- Internamente, lo que ocurre es esto:

- Cuando definimos un método dentro del cuerpo de la clase (fuera del constructor), JavaScript NO lo pone dentro del objeto `this`, sino que lo agrega automáticamente al `prototype` de la clase.

Es decir:

``` javascript

console.log(PersonClass.prototype.showAge); // [Function: showAge]
console.log(pedro.__proto__ === PersonClass.prototype); // true

```

Entonces todos los objetos (`pedro`, `juan`, etc.) comparten la misma función `showAge`, en lugar de duplicarla dentro de cada objeto.

# Características de las clases

## 1. Clases are not hoisted (las clases no son elevadas)

En JavaScript, las **FUNCIONES** declaradas se "hoistean" (se mueven al inicio del contexto de ejecución), lo que significa que puedes llamarlas antes de declararlas:

```javascript

sayHi(); // ✅ funciona

function sayHi() {
  console.log("Hola!");
}

```

## Con clases

### Pero las clases no funcionan así. Si intentamos usarlas antes de declararlas, obtendremos un error:

``` javascript
const p = new Person("Ana"); // ❌ ReferenceError: Cannot access 'Person' before initialization

```

``` javascript

class Person {
  constructor(name) {
    this.name = name;
  }
}

```

- Esto evita usar **clases** antes de definirlas, forzando un código más claro y ordenado.

## 2. Classes are first-class citizens (las clases son ciudadanos de primera clase)

Esto significa que las clases en **JS** son tratadas como cualquier otro valor (igual que funciones).

Podemos:

- Guardarlas en variables
- Pasarlas como argumento a funciones
- Retornarlas desde funciones
- Guardarlas en estructuras de datos (arrays, objetos, etc.)

```javascript

function classFactory(type) {
  if (type === "dog") {
    return class Dog {
      speak() {
        console.log("Guau 🐶");
      }
    };
  } else {
    return class Cat {
      speak() {
        console.log("Miau 🐱");
      }
    };
  }
}

const Animal = classFactory("dog");
const pet = new Animal();
pet.speak(); // "Guau 🐶"

```

Esto hace que las clases sean muy flexibles, igual que las funciones.

## 3. Classes are executed in strict mode (las clases se ejecutan en modo estricto)

El "strict mode" es un modo especial de JS que ayuda a evitar errores silenciosos y malas prácticas. 

Por ejemplo:

- Prohíbe usar variables sin declararlas
- `this` dentro de funciones no referirá al objeto global, sino que será `undefined` (más seguro)
- Prohíbe borrar propiedades que no se pueden borrar (`delete Object.prototype` → error)

⚡ **Las clases siempre se ejecutan en modo estricto automáticamente, incluso si no usas "use strict";.**

Ejemplo:

```javascript

class Test {
  constructor() {
    undeclaredVar = 10; // ❌ ReferenceError en modo estricto
  }
}

new Test();

```

- Esto garantiza más seguridad en el código.

## ✅ En resumen:

- **No hoisting** → no podemos00 usarlas antes de declararlas.

- **First-class citizens** → podemos tratarlas como valores normales (pasarlas, retornarlas, etc.).

- **Strict mode** → se ejecutan siempre bajo reglas más seguras.

# Actualización con las palabras `constructor` +`super()`
 ## Cuando usarla

Nos debemos hacer la siguiente pregunta:

```

¿Necesito inicializar nuevas propiedades o modificar cómo se crea el objeto?

```

Si la respuesta es no, **no necesitamos constructor**.
Si la respuesta es sí, entonces **sí necesitamos** `constructor` + `super()`.

## 1. No necesita la palabra `super()`
Aquí estamos creando un método nuevo en el objeto `Perro` que es `hacerSonido`.

``` javascript

class Animal {
  constructor(nombre) {
    this.nombre = nombre;
  }
}

class Perro extends Animal {
  hacerSonido() {
    console.log("Guau 🐶");
  }
}

const perro = new Perro("Firulais");

```

**JS** hace esto automáticamente:

``` javascript

constructor(...args) {
  super(...args);
}

```
## 2. Caso 2 — SÍ necesitamos `constructor`

Necesitamos `constructor` cuando queremos agregar propiedades nuevas.

Ejemplo:

``` javascript
class Perro extends Animal {
  constructor(nombre, raza) {
    super(nombre);
    this.raza = raza;
  }
}

```
Uso:

``` javascript

const perro = new Perro("Firulais", "Labrador");

```

# 3. Otro caso donde necesitamos `constructor`

Cuando necesitamos hacer lógica extra al crear el objeto.

Ejemplo:

``` javascript

class Perro extends Animal {
  constructor(nombre) {
    super(nombre);

    console.log("Se creó un perro");
  }
}

```

# 4. Pregunta clave

```

Si la clase hija solo agrega o sobrescribe métodos → NO necesita constructor

Si agrega propiedades nuevas → SÍ necesita constructor

```
En otras palabras:
- ¿El constructor del padre ya me sirve?

Si sí → no escribimos `constructor`
Si no → escribimos `constructor` + `super()`

# 5. Resumen

| Situación                  | Constructor |
| -------------------------- | ----------- |
| Solo heredar métodos       | ❌ No        |
| Sobrescribir métodos       | ❌ No        |
| Agregar nuevas propiedades | ✅ Sí        |
| Modificar inicialización   | ✅ Sí        |
