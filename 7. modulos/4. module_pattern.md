# Module Pattern (patrón de módulo)

Es un concepto muy importante en JavaScript, especialmente para entender **cómo se organizaba el código antes de ES6** (antes de `import` y `export`), y también cómo **funcionan internamente los módulos modernos**.

# 1️⃣ ¿Por qué se usaban funciones?

Antes de que existieran los módulos (`export` / `import`) o las clases modernas, **las funciones eran la única forma de crear un ámbito privado en JavaScript**.

Esto es porque cada función crea su propio **Execution Context** (contexto de ejecución) y su propio **Scope**.

# Ejemplo:

``` javascript

function ejemplo() {
  const secreto = "👀 solo dentro de esta función";
  console.log(secreto);
}

ejemplo(); // ✅ se ejecuta
console.log(secreto); // ❌ ReferenceError — no existe fuera

```

Esto significa que **las variables declaradas dentro de una función no son accesibles fuera de ella, a menos que se “expongan” manualmente** (por ejemplo, con `return`).

# 2️⃣ Funciones = Privacidad de datos

Las funciones permiten crear **variables privadas** gracias a los **closures**.

Un [closure](../2.%20funciones/7.%20closures.md) mantiene el acceso a las variables de su entorno aunque la función original ya haya terminado.

# Ejemplo:

``` javascript

function crearContador() {
  let contador = 0; // 🔒 variable privada

  return {
    incrementar() { contador++; },
    obtener() { return contador; }
  };
}

const miContador = crearContador();
miContador.incrementar();
console.log(miContador.obtener()); // 1
console.log(miContador.contador);  // ❌ undefined

```

## ✅ Lo que pasa aquí:

- `contador` vive dentro del scope de `crearContador()`.
- Nadie fuera puede modificar la variable `contador` directamente.

- Las funciones `incrementar()` y `obtener()` mantienen acceso a esa variable (gracias al `closure`).

# 3️⃣ Funciones permiten “exportar” (retornar) datos o métodos públicos

Ya que una función puede devolver valores o funciones, se usa `return` para exponer solo lo necesario (lo público).

# Ejemplo (patrón módulo):

``` javascript

const Carrito = (function() {
  let productos = []; // 🔒 privado

  function agregar(producto) {
    productos.push(producto);
  }

  function listar() {
    console.log(productos);
  }

  // 👇 Exponer solo lo público
  return {
    agregar,
    listar
  };
})();

```
## Así:

- `productos` es privado.

- `agregar()` y `listar()` son públicos, porque se devuelven en el objeto.


# 4️⃣ Entonces, ¿por qué no usar objetos directamente?

- Porque los objetos normales no tienen privacidad.

# Ejemplo:

``` javascript

const persona = {
  nombre: "Bryan",
  edad: 30
};

console.log(persona.nombre); // 👀 cualquiera puede acceder
persona.edad = 99; // 😬 cualquiera puede modificar

```

## En cambio, las funciones:

- Crean su propio ámbito privado.

- Permiten retornar valores controlados.

- Y combinadas con **closures** → permiten **encapsular datos** (ocultarlos del exterior).


### Por eso el Module Pattern clásico se basa en funciones:
### Brinda privacidad, encapsulación, y control de acceso sin necesidad de clases ni export/import.


# 5️⃣ Ahora sí ¿qué es el Module Pattern?

El **Module Pattern** es una forma de organizar y encapsular código en JavaScript para:

- **Evitar contaminar el scope global** (el espacio de nombres global).

- Proteger datos privados usando **closures**.

- **Exponer solo lo necesario** (una “API pública”).

## En pocas palabras:

Es una técnica que permite crear variables privadas y funciones públicas, simulando algo parecido a **clases** o **namespaces**.

# 6️⃣ El problema que resuelve

Antes de que existieran los módulos (`import` / `export`), **todo el código JS compartía el mismo ámbito global**.

# Ejemplo:

``` javascript

// archivo1.js
var contador = 0;

// archivo2.js
var contador = "hola"; // 💥 pisa el valor del otro script

```

- Esto provocaba conflictos de nombres, difícil mantenimiento y errores.

# 7️⃣ Cómo se crea un Module Pattern

El patrón de módulo usa **funciones autoejecutadas (IIFE)** para crear un scope privado.

# 🧠 Sintaxis base:

``` javascript

const miModulo = (function() {
  // 👇 Variables y funciones privadas
  let contador = 0;

  function incrementar() {
    contador++;
  }

  function obtenerContador() {
    return contador;
  }

  // 👇 API pública (lo que "se exporta")
  return {
    incrementar,
    obtenerContador
  };
})();

```

# 🧪 Uso:

``` javascript

miModulo.incrementar();
console.log(miModulo.obtenerContador()); // 1
console.log(miModulo.contador); // ❌ undefined (privado)

```

# ✅ Lo importante:

- Todo lo que está **dentro de la IIFE** es privado.

- Solo lo que se **retorna es público**.

- Esto **simula un módulo**, incluso antes de ES6.

# 🧩 8️⃣ Relación con los módulos modernos (ES Modules)

Hoy en día, **ya no es necesario usar IIFE** gracias a los módulos nativos (`import` / `export`).

# Por ejemplo:

``` javascript

export const productos = [];
export function agregarProducto(nombre) { productos.push(nombre); }

```
# Y esto:

``` javascript

import { agregarProducto } from './carrito.js';

```

## ➡️ Hace lo mismo, pero de forma nativa y más legible.
- Sin embargo, el **module pattern** sentó las bases para cómo funcionan los módulos modernos en JS.

# 🧩 9️⃣ Variantes del Module Pattern

Estas son versiones antiguas cuando no existían los módulos de forma nativa en JavaScript.

(Es problame que aún exista código con alguna de extas sintáxis)

- Hay algunas versiones comunes:

| Nombre                       | Descripción                                                                             |
| ---------------------------- | --------------------------------------------------------------------------------------- |
| **Revealing Module Pattern** | Retorna un objeto que expone funciones públicas con los mismos nombres que las privadas |
| **Singleton Module**         | Garantiza que solo exista una única instancia del módulo                                |
| **CommonJS / AMD**           | Son implementaciones previas a ES Modules (Node.js y navegadores antiguos)              |
