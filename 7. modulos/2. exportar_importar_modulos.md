# 1. Los módulos son lo primero que se ejecuta

Cuando se importa un módulo en otro archivo, ese módulo se evalúa primero antes de seguir con el código del archivo principal.

``` javascript

// archivo: util.js
console.log("Se ejecuta util.js");
export const x = 10;

// archivo: app.js
import { x } from './util.js';
console.log("Se ejecuta app.js");

```

``` javascript

Se ejecuta util.js
Se ejecuta app.js

```

# EXPORT

- Significa que esa función/variable/clase será pública y exportada al exterior.

- Se puede escribir al momento de declararla:

``` javascript

export function suma(a, b) {
  return a + b;
}

export const pi = 3.1416;

```

O se puede declarar normalmente y exportarla después:

``` javascript

const carro = "Toyota";
const casa = "Pequeña";

export { carro, casa };

```

# 2. Se pueden exportar varias variables juntas al final

``` javascript

const perro = "Firulais";
const gato = "Michi";

export { perro, gato };

```

## Esto se llama `exportación nombrada (named export)`.

## Y luego puedes importarlas así:

``` javascript

import { perro, gato } from "./animales.js";

console.log(perro, gato); // Firulais Michi

```

# 3. Alias al exportar

Se puede cambiar el nombre cuando se **define en el módulo**:

``` javascript

const carro = "Toyota";
const casa = "Pequeña";

export { carro as coche, casa as hogar };

```

- Aquí se exporta `carro` con el nombre `coche` y `casa` como `hogar`.

Y al importar:

``` javascript

import { coche, hogar } from "./modulo.js";

console.log(coche); // Toyota
console.log(hogar); // Pequeña

```

# 4. Alias al importar

También se puede dejar los nombres iguales en el módulo, pero **renombrarlos al importarlos**:

``` javascript

// modulo.js
export const perro = "Firulais";
export const gato = "Michi";

```

``` javascript

// app.js
import { perro as dog, gato as cat } from "./modulo.js";

console.log(dog); // Firulais
console.log(cat); // Michi

```

# 5. Alias con `export default`

Si se usa un `export default`, se puede importar con cualquier nombre, sin necesidad de `as`.

``` javascript

// modulo.js
export default function saludar() {
  console.log("Hola!");
}

```

``` javascript

// app.js
import saludo from "./modulo.js";

saludo(); // Hola!

```

- Aquí da igual si se llama `saludo`, `hi`, o `pepito`.


## Entonces sí:

- Alias al exportar con `as`.

- Alias al importar con `a`s.

- Con `export default` el alias es libre al importar.

# 6. `namespace import` en ES Modules

Con `import * as Alias from 'archivo.js'` se importa todo lo exportado de ese módulo dentro de un objeto.

## Ejemplo

Supongamos que hay un módulo `script1.js`:

``` javascript

// script1.js
export const carro = "Toyota";
export const casa = "Grande";
export function saludar() {
  console.log("Hola desde script1!");
}

```

Ahora se importa así:

``` javascript

// app.js
import * as ScriptNuevo from "./script1.js";

console.log(ScriptNuevo.carro);   // Toyota
console.log(ScriptNuevo.casa);    // Grande
ScriptNuevo.saludar();            // Hola desde script1!

```

Aquí `ScriptNuevo` es un objeto que contiene todas las exportaciones nombradas de `script1.js`.

# ¿Y qué pasa con `export default?`

Si en `script1.js` tenemos:

``` javascript

export default function principal() {
  console.log("Soy la función principal");
}

```

- Ese default se vuelve la propiedad `.default` del objeto:

``` javascript

import * as ScriptNuevo from "./script1.js";

ScriptNuevo.default(); // Soy la función principal

```

# ✅ Cuándo usarlo

- Cuando queremos agrupar todo un módulo bajo un mismo nombre.

- Muy útil si tenemos un módulo con muchas funciones/constantes.

- También si queremos evitar colisiones de nombres en tu archivo.

# 7. Diferencia entre Default Export y Named Export

## 1. Named Export

- Exporta una o varias variables/funciones con un nombre fijo.

- Al importar, debes usar el mismo nombre (a menos que uses un alias).

``` javascript

// modulo.js
export const carro = "Toyota";
export function saludar() {
  console.log("Hola!");
}

```

``` javascript

// app.js
import { carro, saludar } from "./modulo.js";

console.log(carro); // Toyota
saludar();          // Hola!

```

- Se Puede exportar varias cosas en el mismo archivo.

- El nombre es obligatorio y debe coincidir.

## 2. Default Export

- Cada archivo puede tener solo un `export default`.

- Al importar, se puede llamar como sea.

``` javascript

// modulo.js
export default function() {
  console.log("Soy la función principal");
}

```

``` javascript

// app.js
import loQueYoQuiera from "./modulo.js";

loQueYoQuiera(); // Soy la función principal

```

- Ideal para exportar una única cosa principal de un módulo.
- El nombre es libre al importar.

## Diferencias clave

| Característica       | Named Export                   | Default Export               |
| -------------------- | ------------------------------ | ---------------------------- |
| Cantidad por archivo | Varias                         | Solo 1                       |
| Nombre al importar   | Obligatorio (o con `as`)       | Libre                        |
| Sintaxis             | `export const a = ...`         | `export default function...` |
| Uso común            | Librerías con muchas funciones | Exportar “lo principal”      |


# Importaciones NO son una copia ⚡

## Esto es muy importante 👇

Cuando hacemos:

``` javascript

// modulo.js
export let contador = 0;
export function incrementar() {
  contador++;
}

```

``` javascript

// app.js
import { contador, incrementar } from "./modulo.js";

console.log(contador); // 0
incrementar();
console.log(contador); // 1 ✅

```

- Lo que se importa **no es una copia**, es una **referencia al valor vivo** del módulo.

- Es decir, si el módulo cambia la variable, ese cambio se refleja donde se importó.

- Por eso se dice que las importaciones en JS son **live bindings (enlazadas en vivo)**.

## Resumen:

- `Default export` → solo uno, nombre libre al importar.

- `Named export` → varios, nombre fijo al importar.

- `Imports` no son copias → son referencias vivas al módulo original.

# Cómo las importaciones no son copias, sino referencias vivas (live bindings).

## 🧩 Estructura de archivos

Tendremos 3 archivos:

``` javascript

contador.js
modificar.js
ver.js

```

## 1. `contador.js` → el módulo base

``` javascript

// contador.js
export let contador = 0;

export function incrementar() {
  contador++;
}

export function decrementar() {
  contador--;
}

```
Aquí exportamos una variable `(contador)` y dos funciones que la modifican.

## 2. `modificar.js` → cambia el valor

``` javascript

// modificar.js
import { incrementar, decrementar } from "./contador.js";

export function acciones() {
  incrementar();
  incrementar();
  decrementar();
}

```

Este módulo usa las funciones para modificar el valor de `contador`.

## 3. `ver.js` → observa el valor

``` javascript

// ver.js
import { contador } from "./contador.js";
import { acciones } from "./modificar.js";

console.log("Valor inicial:", contador); // 0
acciones();
console.log("Después de modificar:", contador); // 1 ✅

```

## 🧠 Explicación

- Cuando `ver.js` importa `contador`, no obtiene una copia, sino una **referencia viva al valor original** de `contador` en `contador.js`.

- `modificar.js` modifica el mismo `contador`, y cuando `ver.js` lo imprime de nuevo, el cambio ya se refleja.

## Esto ocurre porque los módulos ES de JavaScript son singletons:
## Se cargan una sola vez en memoria, y todos los que los importan acceden al mismo módulo compartido.

## ⚙️ Resultado al ejecutar `ver.js`

``` javascript

Valor inicial: 0
Después de modificar: 1

```

## 💡 Importante

- Si queremos tener copias independientes, tendríamos que clonar los valores manualmente, por ejemplo:

``` javascript

let copia = contador;

```

Pero por defecto, los módulos de ES mantienen un estado **compartido entre todos los imports**.
